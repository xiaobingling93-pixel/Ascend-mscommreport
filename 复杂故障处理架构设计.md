# 复杂故障处理架构设计

## 1. 背景和目标

### 1.1 背景

当前故障检测系统中存在两类故障：

- **简单故障**：规则可配置化，通过 YAML 配置文件即可处理
- **复杂故障**：需要根据上下文进行多条件决策，每种复杂故障可能有多种不同解决方案

复杂故障的特点：
- 需要综合多个信息源（日志、通信域、网卡信息等）
- 决策逻辑复杂，有优先级
- 每次只输出一个解决方案（优先级最高的）
- 决策粒度：通信域级别

### 1.2 目标

设计一个可扩展、高内聚低耦合的复杂故障处理架构：

1. **扩展性**：易于添加新的决策规则和解决方案
2. **可读性**：规则和解决方案清晰对应
3. **可维护性**：避免大量 if-else 语句
4. **优先级管理**：集中的优先级配置
5. **按需收集**：信息按需从日志中提取
6. **规则隔离**：每个规则处理后清空缓存，避免相互影响

## 2. 整体架构

```

│                     FaultAnalyzer                            │
│                     (故障分析器)                              │
│  - _process_complex_faults() 方法按故障类型分发               │
└───────────────────────────┬─────────────────────────────────┘
                            │
            ┌───────────────┼───────────────┬──────────────┐
            ▼               ▼               ▼              ▼
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │RankNotConn.  │ │ParamPlaneLink│ │ExecTimeout   │ │ExecDiff      │
    │DecisionEngine│ │DecisionEngine│ │Engine        │ │Engine        │
    └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
           │                │                │                │

    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ 9 Rules      │ │ 13 Rules     │ │ 4 Rules      │ │ 1 Rule       │
    │ + Collectors │ │ + RuleBase   │ │ + Collectors │ │ + Collectors │
    └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

            ┌─────────────────────────────────────────────────────────────┐
            │                      FaultContext                           │
            │                      (故障分析上下文)                         │
            │  - faults                      原始故障列表                  │
            │  - comm_info_map               通信域信息映射                │
            │  - comm_domain_process_map      通信域→进程号映射            │
            │  - fault_groups                去重后的故障分组             │
            │  - log_files                   日志文件列表                 │
            │  - extended_info               扩展信息/缓存                │
            └─────────────────────────────────────────────────────────────┘
```

**处理流程**：

1. `FaultAnalyzer._process_complex_faults()` 遍历 `context.fault_groups`
2. 根据 `category.level3` 分发到对应的 `DecisionEngine`
3. `DecisionEngine` 按优先级顺序尝试匹配规则，第一个匹配的规则执行 `apply()`
4. 每个规则处理后清空 `context.extended_info` 缓存

## 4. 核心类设计

### 4.1 DecisionRule（决策规则基类）

抽象基类，每个规则继承此类并实现 `match`、`generate_solution` 和 `apply` 方法。

```python
class DecisionRule(ABC):
    """
    决策规则基类

    每个规则继承此类并实现 match 和 generate_solution 方法。
    信息收集逻辑直接在 match 或 generate_solution 方法中实现。

    设计原则：
    - 规则和解决方案强绑定，放在同一个类中
    - match 方法负责判断是否匹配（可包含信息收集逻辑）
    - generate_solution 方法负责生成解决方案
    - 可以将收集到的信息缓存到 context.extended_info
    - 优先级通过构造函数传入，数值越小优先级越高
    """

    def __init__(self, priority: int):
        self._priority = priority

    @property
    def priority(self) -> int:
        return self._priority

    @abstractmethod
    def match(self, context: FaultContext, key: str) -> bool:
        """判断是否匹配该规则"""
        pass

    @abstractmethod
    def generate_solution(self, context: FaultContext) -> List[str]:
        """生成解决方案文本列表"""
        pass

    @abstractmethod
    def apply(self, context: FaultContext, key: str) -> None:
        """应用规则到上下文，直接修改 context.fault_groups"""
        pass
```

### 4.2 ParamPlaneLinkEstablishRule（参数面建链规则基类）

继承自 `DecisionRule`，为参数面建链超时故障的规则提供公共逻辑。

```python
class ParamPlaneLinkEstablishRule(DecisionRule):
    """
    参数面建链超时故障规则基类

    提供两个核心公共方法：
    - extract_param_plane_fault_info: 提取故障组、identifier、匹配故障、rank对
    - trace_dest_rank_chain: 递归追踪 destRank 链路，获取最终故障源头
    """

    def extract_param_plane_fault_info(
        self, context: FaultContext, key: str
    ) -> Optional[Tuple[FaultGroup, str, List[FaultInstance], List[Tuple[int, int]]]]:
        """
        提取参数面建链超时故障的公共信息

        1. 获取当前故障组并验证 level3 == "param_plane_link_establish_timeout"
        2. 从故障组中提取 identifier
        3. 筛选所有 param_plane_link_establish_timeout 类型的故障
        4. 收集所有故障的 rank_pairs（通过 RankPairCollector）

        Returns: (fault_group, identifier, matching_faults, rank_pairs) 或 None
        """

    def trace_dest_rank_chain(
        self, context: FaultContext, identifier: str,
        initial_dest_rank: int, visited: Set[int]
    ) -> Optional[Tuple[int, int, str, FaultInstance]]:
        """
        递归追踪 destRank 链路

        逻辑：
        1. 根据通信域 id 和 destRank 获取进程号 key
        2. 判断该进程号 key 是否有参数面建链超时故障
        3. 如果有，获取该故障的 rank_pairs，更新 srcRank 和 destRank
        4. 递归判断新的 destRank
        5. 直到 destRank 没有参数面建链超时故障或节点已访问（避免循环）

        Returns: (final_src_rank, final_dest_rank, final_identifier, fault_instance) 或 None
        """
```

### 4.3 FaultContext（故障分析上下文）

数据容器，存储分析过程中的所有数据。

```python
@dataclass
class FaultContext:
    """
    故障分析上下文

    存储分析过程中的所有信息，包括：
    - faults: 原始故障列表（由 FaultDetector 生成）
    - comm_info_map: 通信域信息映射，key 为组合键 worker_id|process_id，value 为通信域信息列表
    - comm_domain_process_map: 通信域到进程号的映射，key 为 identifier|rankId，value 为组合键
    - fault_groups: 去重后的故障分组（由 FaultDeduplicator 生成）
    - log_files: 日志文件列表
    - extended_info: 扩展信息字典，用于存储规则收集的额外信息或缓存结果

    faults: List[FaultInstance] = field(default_factory=list)
    comm_info_map: Dict[str, List[CommunicationInfo]] = field(default_factory=dict)
    comm_domain_process_map: Dict[str, str] = field(default_factory=dict)
    fault_groups: Dict[str, 'FaultGroup'] = field(default_factory=dict)
    log_files: List[LogFile] = field(default_factory=list)
    categories: List[FaultCategory] = field(default_factory=list)
    statistics: Optional['FaultStatistics'] = None
    extended_info: Dict[str, Any] = field(default_factory=dict)

    # 扩展信息操作
    def set(self, key: str, value: Any) -> None: ...
    def get(self, key: str, default: Any = None) -> Any: ...
    def has(self, key: str) -> bool: ...

    # 通信域查询
    def get_comm_info(self, identifier: str, rank_id: int) -> Optional[CommunicationInfo]: ...
    def get_process_id(self, identifier: str, rank_id: int) -> Optional[str]: ...
    def get_worker_id(self, identifier: str, rank_id: int) -> Optional[str]: ...

    # 日志文件路径查询
    def get_run_plog_path(self, identifier: str, rank_id: int) -> List[str]: ...
    def get_debug_plog_path(self, identifier: str, rank_id: int) -> List[str]: ...
    def get_run_device_log_path(self, identifier: str, rank_id: int) -> List[str]: ...
    def get_debug_device_log_path(self, identifier: str, rank_id: int) -> List[str]: ...
```

### 4.6 DecisionEngine（决策引擎）

每个复杂故障类型都有一个决策引擎，管理所有决策规则，按优先级执行匹配。

```python
class XxxDecisionEngine:
    """
    故障决策引擎

    管理所有决策规则，按优先级顺序执行匹配。
    优先级数值越小，优先级越高。
    """

    def __init__(self):
        self._rules: List[DecisionRule] = [
            RuleA(priority=10),
            RuleB(priority=20),
            RuleC(priority=30),
        ]
        self._rules.sort(key=lambda rule: rule.priority)

    def process(self, context: FaultContext, key: str) -> None:
        """
        处理复杂故障，直接修改 context.fault_groups

        每个规则处理后都会清空 extended_info 缓存，避免规则间相互影响。
        """
        for rule in self._rules:
            if rule.match(context, key):
                rule.apply(context, key)
                context.extended_info.clear()
                return
            else:
                context.extended_info.clear()
```

**职责**：
- 管理所有决策规则
- 集中配置优先级（通过构造函数）
- 按优先级顺序执行规则匹配
- 每个规则处理后清空缓存，避免规则间相互影响

## 5. 故障分发机制

`FaultAnalyzer._process_complex_faults()` 负责将故障组分发到对应的决策引擎。采用**单次遍历**的方式，在循环内部根据 `level3` 分发到对应的引擎，时间复杂度 O(n)：

```python
def _process_complex_faults(self, context: FaultContext) -> None:
    rank_not_connected_engine = RankNotConnectedDecisionEngine()
    param_plane_engine = ParamPlaneLinkEstablishDecisionEngine()
    exec_timeout_engine = ExecTimeoutEngine()

    # 单次遍历 fault_groups，按 level3 分发
    for key, group in list(context.fault_groups.items()):
        level3 = group.category.level3
        if level3 == FAULT_RANK_NOT_CONNECTED:
            rank_not_connected_engine.process(context, key)
        elif level3 == FAULT_PARAM_PLANE_LINK_ESTABLISH_TIMEOUT:
            param_plane_engine.process(context, key)
        elif level3 == FAULT_NOTIFY_WAIT_TIMEOUT:
            exec_timeout_engine.process(context, key)

    # 检查集群HCCL配置一致性（特殊：不基于故障组分发）
    exec_diff_engine = ExecDiffEngine()
    exec_diff_engine.process(context, FAULT_HCCL_CONFIG_DIFF_CHECK)
```

**性能说明**：三种故障类型按不同的 `level3` 分发，互不影响（`rank_not_connected` 可能删除的故障组属于其他 `level3`），因此合并为单次遍历是安全的。`ExecDiffEngine` 使用特殊 key，不属于 fault_groups 遍历，保持独立调用。

**分发规则**：

| level3 值（常量） | 决策引擎 | 说明 |
|-----------|---------|------|
| `FAULT_RANK_NOT_CONNECTED` | `RankNotConnectedDecisionEngine` | 部分rank未连接到server节点 |
| `FAULT_PARAM_PLANE_LINK_ESTABLISH_TIMEOUT` | `ParamPlaneLinkEstablishDecisionEngine` | 参数面建链超时 |
| `FAULT_NOTIFY_WAIT_TIMEOUT` | `ExecTimeoutEngine` | 算子执行超时 |
| `FAULT_HCCL_CONFIG_DIFF_CHECK` | `ExecDiffEngine` | 集群HCCL配置一致性（特殊：不基于故障组分发） |

> 所有 level3 常量定义在 `detector/fault_constants.py` 中，与 `fault_config.yaml` 的配置保持一致。

## 8. 故障去重机制

`FaultDeduplicator` 负责对故障进行去重：

1. **去重键生成**：`{level1}.{level2}.{level3}|{identifier}`
2. **通信域信息合并**：只取 `fault_list[0]` 的通信域信息
3. **按 level3 限制**：同一 level3 分类下超过1个故障组时，只保留发生时间最早的1个

## 9. 扩展指南

### 9.1 添加新的决策规则

1. 在对应故障类型的 `rules/` 目录下创建新规则类，继承 `DecisionRule`（或 `ParamPlaneLinkEstablishRule`）
2. 实现 `match`、`generate_solution` 和 `apply` 方法
3. 在 `DecisionEngine.__init__` 中注册规则并设置优先级

```python
# rules/new_rule.py
class NewRule(DecisionRule):
    """新规则"""

    def __init__(self, priority: int = 15):
        super().__init__(priority=priority)

    def match(self, context: FaultContext, key: str) -> bool:
        # 判断逻辑
        return True/False

    def generate_solution(self, context: FaultContext) -> List[str]:
        # 生成解决方案
        return ["解决方案文本"]

    def apply(self, context: FaultContext, key: str) -> None:
        if key in context.fault_groups:
            solutions = self.generate_solution(context)
            context.fault_groups[key].solution = "\n".join(solutions)

# decision_engine.py
self._rules: List[DecisionRule] = [
    ExistingRule(priority=10),
    NewRule(priority=15),           # 新规则，优先级15
    ExistingRule(priority=20),
]
```

### 9.2 添加新的信息收集器

1. 在对应故障类型的 `collectors/` 目录下创建新收集器类
2. 实现提取方法
3. 在规则中调用收集器

```python

    """新信息收集器"""

    @staticmethod
    def extract(source_file: str, **kwargs) -> Any:
        # 从日志文件中提取信息
        pass

# rules/new_rule.py
class NewRule(DecisionRule):
    def match(self, context: FaultContext, key: str) -> bool:
        # 按需收集信息
        info = NewCollector.extract(source_file)
        # 判断逻辑...
        return True/False
```

### 9.3 添加新的复杂故障类型

1. 在 `processors/` 下创建新的子目录
2. 在 `fault_constants.py` 中添加新的 level3 常量
3. 实现 DecisionEngine、Rules 和 Collectors
4. 在 `FaultAnalyzer._process_complex_faults()` 的单次遍历中注册新引擎

```python
# processors/new_fault/
├── __init__.py
├── decision_engine.py
├── collectors/
│   ├── __init__.py
│   └── new_collector.py
└── rules/
    ├── __init__.py
    ├── rule_a.py
    └── rule_b.py

# fault_constants.py 新增
FAULT_NEW_TYPE = "new_fault_type"

# fault_analyzer.py _process_complex_faults() 中新增 elif 分支
for key, group in list(context.fault_groups.items()):
    level3 = group.category.level3
    if level3 == FAULT_RANK_NOT_CONNECTED:
        rank_not_connected_engine.process(context, key)
    elif level3 == FAULT_PARAM_PLANE_LINK_ESTABLISH_TIMEOUT:
        param_plane_engine.process(context, key)
    elif level3 == FAULT_NOTIFY_WAIT_TIMEOUT:
        exec_timeout_engine.process(context, key)
    elif level3 == FAULT_NEW_TYPE:
        new_fault_engine.process(context, key)
```

## 10. 设计优势

1. **规则即解决方案**：每个 Rule 类同时包含匹配逻辑和解决方案，逻辑内聚
2. **优先级集中管理**：所有规则优先级在 DecisionEngine 一目了然
3. **避免大量 if-else**：每个规则是独立类，易于维护
4. **按需信息收集**：
   - 收集器按需从日志文件中提取信息
   - 提取的数据可缓存到 `context.extended_info`
   - 不同规则可以从不同文件中提取各自需要的信息
5. **规则隔离**：每个规则处理后清空缓存，避免规则间相互影响
6. **易于测试**：每个规则和收集器可独立测试
7. **易于扩展**：新增规则只需添加新类并注册
8. **跨故障组判断**：支持需要跨多个故障组进行判断的复杂场景（如网卡不一致）
9. **链路追踪**：`trace_dest_rank_chain` 支持递归追踪故障传播链路，定位最终故障源头
10. **规则基类复用**：`ParamPlaneLinkEstablishRule` 封装公共逻辑，减少重复代码
11. **常量集中管理**：`fault_constants.py` 统一定义 level3 字符串常量，避免硬编码散落各处，修改时不易遗漏
12. **单次遍历分发**：`_process_complex_faults()` 通过单次遍历按 level3 分发到对应引擎，O(n) 时间复杂度
